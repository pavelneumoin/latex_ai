# Интеграция WorksheetAI ↔ Obsidian vault (Lessons)

**Статус:** черновик для решения утром 2026-05-27. Реализация не начинается до подтверждения.

---

## 1. Контекст и цель

У пользователя уже несколько лет копится Obsidian-vault в `E:\YA\YandexDisk\Lessons\vault\` (~10 папок, ~1-2 GB markdown + картинки). Внутри: атомарные методические карточки по математике/информатике, индекс готовых уроков из `library/`, импортированные банки задач из ФИПИ, sdamgia, kompege, mathege. Это уже проверенный «человеком» материал. Цель — использовать vault как первичный источник задач/материалов для LLM-генератора рабочих листов в WorksheetAI, чтобы (а) сильно снизить расход токенов на «придумай задачу», (б) повысить качество за счёт работы с готовыми проверенными задачами, (в) не дублировать многолетнюю накопленную базу.

## 3. Варианты доступа

Оцениваем плюсы/минусы. Финальное решение — утром.

### A. Прямой парсинг markdown в реальном времени

**Суть:** на запрос пользователя бэкенд ходит по vault по имени папки/тегу, читает `.md`-файлы, парсит frontmatter, возвращает первые N подходящих карточек. Без индексов и embedding.

**Плюсы:**
- Ноль инфраструктуры. Просто `glob` + `frontmatter.load()`.
- Никаких векторов, embedder, БД.
- Прозрачно: легко отлаживать, легко добавить новые теги.

**Минусы:**
- Не масштабируется (8000+ задач, на запрос «производная» придётся читать тысячи файлов).
- Только точные совпадения по тегам/полям, никакой семантики.
- Зависит от того, что vault лежит локально (на сервере придётся синкать).

**Оценка работ:** S (1-2 дня).

**Что нужно:** скрипт `vault_reader.py` (glob + frontmatter) + соглашение о тегах/папках.

---

### B. Векторная база (Qdrant / Chroma / pgvector) + локальный embedder

**Суть:** один раз индексируем vault → векторизуем условия и теорию через локальный embedder (e5-small, multilingual-MiniLM, BGE-M3) → семантический поиск top-K → подмешиваем в промпт LLM.

**Плюсы:**
- Semantic search: «задачи на движение по реке» найдёт и «лодка плывёт по течению».
- Хорошее качество результата.
- Один раз настроил — работает.

**Минусы:**
- Инфраструктура: либо Qdrant контейнером, либо Chroma локально, либо pgvector в Postgres.
- Embedder весит 100-500 МБ (или API-вызовы).
- Реиндексация при добавлении карточек (cron / file watcher).
- Сложно для дебага: непонятно, почему конкретная задача попала/не попала в top-K.

**Оценка работ:** L (5-7 дней с инфрой).

**Что нужно:** Qdrant (Docker) ИЛИ Chroma, sentence-transformers, скрипт индексатора, скрипт ретривера, обновление при изменении vault.

---

### C. Локальный full-text (Meilisearch / Tantivy / Postgres TSVECTOR)

**Суть:** индексируем vault обычным full-text поиском (без семантики). Учитель ищет «производная сложной» — получает все карточки, где встречаются эти слова.

**Плюсы:**
- Быстро (Meilisearch особенно).
- Без embedder, без векторов.
- Дёшево и просто в эксплуатации.

**Минусы:**
- Нет семантики («движение по реке» не найдёт «течение реки»).
- Хуже работает с короткими карточками-задачами (мало контекста для full-text).
- Русская морфология требует настройки (стемминг, стоп-слова).

**Оценка работ:** M (3-4 дня).

**Что нужно:** Meilisearch (легко), скрипт индексатора, обвязка для запросов.

---

### D. MCP-сервер для vault

**Суть:** написать MCP-сервер с тулзами `vault_search`, `vault_read_card`, `vault_list_by_tag`. LLM сама во время генерации листа решает, что ей нужно из vault, и вызывает тулзы.

**Плюсы:**
- LLM сама управляет контекстом — оптимальный набор задач для конкретного запроса.
- Переиспользуется в Cowork/Claude Code напрямую.
- Очень хорошо стыкуется с уже существующими agent-пайплайнами пользователя.

**Минусы:**
- **Требует, чтобы LLM-провайдер поддерживал MCP**. Claude — да. GigaChat — нет (пока). DeepSeek — нет. Если основной провайдер не Claude — бесполезно.
- Дороже по токенам (LLM делает несколько round-trip'ов).
- Не получится использовать в простой API-генерации.

**Оценка работ:** M (3-4 дня).

**Что нужно:** Python MCP-сервер (FastMCP / anthropic-skills:mcp-builder), 3-5 тулзов, конфиг для LLM-клиента.

---

### E. SQLite-индекс по vault

**Суть:** один раз парсим vault → строим SQLite с таблицами `cards(id, path, title, subject, class, theme, difficulty, body)`, `tags(card_id, tag)`. Запросы из бэкенда — обычным SQL.

**Плюсы:**
- Очень дёшево, ноль инфраструктуры (SQLite встроен в Python).
- Быстро (миллионы строк — без проблем).
- Можно фильтровать по любому полю frontmatter.
- Файл `vault.sqlite` (~50-200 МБ) легко перевозить между машинами.

**Минусы:**
- Нет semantic (только точные совпадения по тегам/значениям).
- Реиндексация — полная или ручная пометка изменений.

**Оценка работ:** S-M (2-3 дня).

**Что нужно:** скрипт `build_vault_index.py` (один раз), `vault_query.py` (запросы), cron для перестроения.

---

### F. Гибрид B+E (SQLite фильтр + векторка ранжирование)

**Суть:** SQLite — для жёсткой фильтрации (`subject=math AND class=11 AND theme='derivative'`). Векторка — для семантического ранжирования внутри уже отфильтрованного подмножества (10-100 карточек).

**Плюсы:**
- Лучшее из двух миров: точная фильтрация + семантическое попадание.
- Векторка работает на маленьком подмножестве → быстро, дёшево по памяти.
- Если нет векторки — можно деградировать к чистому SQLite (graceful degradation).
- Хорошо масштабируется: vault растёт — SQL ускоряет, векторка не страдает.

**Минусы:**
- Самая сложная архитектура: два хранилища, два индекса.
- Дольше всех в реализации.

**Оценка работ:** L (5-7 дней).

**Что нужно:** SQLite-индекс из (E) + векторка из (B) поверх него.

## 4. Способы заливки vault на сервер

После выбора варианта, vault должен попасть на бэкенд WorksheetAI. Варианты:

| Способ | Плюсы | Минусы |
|---|---|---|
| **`git push` из vault** (если vault уже в git) | Версионирование, дельты, простота. | Vault большой (1-2 GB), картинки раздуют репо. LFS усложняет setup. |
| **rsync/scp по cron каждые N часов** | Простой, инкрементальный, не нужен git в vault. | Нужен ssh-ключ на сервер, cron на ноутбуке. |
| **Yandex Disk Sync на сервере** | Vault уже там, sync «бесплатный». | Зависимость от YD-клиента на сервере (Linux-версия неофициальная). Задержка sync. |
| **Однократная заливка zip + ручное обновление** | Самый простой запуск. | Не обновляется автоматически. Подходит для MVP. |

**Рекомендация по заливке:** на старте — `zip` (однократно), как только индексатор будет стабилен — `rsync` по cron каждый час. Git — только если vault и так под git.

## 5. Рекомендация

**Выбираем вариант F (Гибрид SQLite + векторка)** — с поэтапной реализацией:

1. **Этап 1 (MVP, 2-3 дня):** только SQLite-индекс из (E). Уже даст 80% пользы: фильтрация по предмету/классу/теме/сложности. Промпты `generate_from_material.md` получают список задач через SQL и подмешивают в `{{material}}`.
2. **Этап 2 (опционально, +3-4 дня):** поверх SQLite добавить локальную векторку (Chroma встроенный, без отдельного контейнера) для семантического ранжирования. Включается флагом.

Обоснование: SQLite даёт быстрый MVP без инфраструктуры, его легко перевозить (один файл) и легко дебажить. Векторка добавляется когда станет ясно, что чисто фильтрационного поиска не хватает. Никакой потери: всё, что построено на этапе 1, останется при добавлении этапа 2.

**Заливка:** на MVP — однократный `zip` + `scp`. После пилота — `rsync` по cron.

## 6. Что НЕ делаем этой ночью

Явный список, чтобы не было соблазна сделать больше:

- Не реализуем индексатор vault (ни SQLite, ни векторку).
- Не ставим Qdrant / Chroma / Meilisearch.
- Не пишем парсер frontmatter.
- Не пишем MCP-сервер.
- Не настраиваем синхронизацию vault на сервер.
- Не меняем структуру vault и frontmatter.

**Делаем только этот документ + промпты в `cli/prompts/`** — они написаны так, что одинаково работают и без vault (LLM генерирует сама), и с vault (vault-материал подаётся в `{{material}}`).

## 7. Открытые вопросы для пользователя утром

1. **Какой вариант (A-F) выбираем?** Моя рекомендация — F с поэтапной реализацией, начиная с этапа 1 (SQLite).
2. **Какой LLM-провайдер основной?** Это влияет на вариант D (MCP) и на бюджет токенов. Если в пайплайне будут и Claude, и GigaChat — выбираем модель-нейтральный путь (E/F, не D).
3. **Заливать vault как git или через rsync?** Зависит от того, под git ли vault сейчас.
4. **Embedder локально или через API?** Варианты:
   - локально: e5-small (~120 МБ, CPU-friendly), BGE-M3 (~2 ГБ, нужна GPU для скорости).
   - API: Sber EmbeddingsR, OpenAI text-embedding-3-small, Yandex GPT Embeddings.
   - На MVP вообще без embedder, чисто SQLite.
5. **Размер top-K для подмешивания в промпт?** Стандартное значение 5-10 задач/карточек. Хочется ли учителю видеть, какие карточки подмешались (UI-вопрос)?
6. **Кешировать ли результаты поиска?** Если учитель повторяет «производная 11 класс» десять раз в неделю — кеш экономит. Если все запросы уникальны — нет смысла. На MVP — пропускаем; добавим, если жалобы на скорость.
7. **Какие подразделы vault реально нужны для генератора?** Точно `02-Банк_задач/`. Карточки `08-Математика/`, `09-Информатика/` — для теоретических листов или нет? Это определит размер индекса и фильтры.
8. **Где хранить vault.sqlite?** На сервере WorksheetAI или в самой папке vault (отдельный файл рядом)? Влияет на синхронизацию.
